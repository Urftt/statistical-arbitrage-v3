<objective>
Fix the `lookback_window` grid-search bug where every cell is reported as `blocked` and the heatmap is empty whenever `lookback_window` is used as an optimization axis. The root cause is that `StrategyParameters.model_copy(update=...)` in Pydantic v2 does not re-run field validators, so a float value from `_build_axis_values()` (e.g., `20.0`) is assigned directly to an `int` field, then `range(lookback_window - 1, n)` in `build_rolling_strategy_data()` raises `TypeError: 'float' object cannot be interpreted as an integer`. The same bug exists in `run_walk_forward()` where `best_params` (a `dict[str, float]`) is applied via `model_copy(update=...)` to build the test-window params.

Purpose: Unblock the Optimize tab for `lookback_window` sweeps (the primary reason a user runs grid search) and prevent regression by adding targeted tests that exercise int-typed axes through both `run_grid_search()` and `run_walk_forward()`.

Output:
- Two-line fix in `optimization.py` and `walkforward.py` replacing `model_copy(update=...)` with `StrategyParameters.model_validate({**base_params.model_dump(), **overrides})`.
- Regression tests in `tests/test_optimization.py` and `tests/test_walkforward.py` that exercise `lookback_window` as an axis and assert non-blocked cells/folds.
</objective>

Pydantic v2 behaviour (relevant for the fix):
- `model_copy(update=...)` bypasses validators — assigns values directly.
- `model_validate({...})` runs full validation, and lax-mode int validator coerces
  whole-number floats to int (e.g., `20.0` -> `20`). This is the documented safe path.
